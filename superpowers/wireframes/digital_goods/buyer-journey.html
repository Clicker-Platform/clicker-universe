Notes

  • Two entry points, one login. Entry A (first-time purchase) flows 1→2→3→4→5→6b. Entry B (returning buyer to library) flows 3→6a→6b. The login route is the same — the ?next= query param controls where the buyer lands after verifying.
  • Auth happens once per 7 days. After step 3, the __session cookie lasts 7 days. Steps 4 / 5 / 6 just re-verify it server-side.
  • Buyer identity is module-scoped. Buyer doc lives only at sites/{siteId}/modules/digital_goods/buyers/{uid}. No coupling to the loyalty module or the team-RBAC members collection.
  • Tenant confirmation is the human bottleneck. Step 5 can stall indefinitely if the tenant doesn't act. Plan 3 adds reminders.
  • Two distinct auth helpers. Buyer endpoints use adminAuth.verifySessionCookie() directly. Admin endpoints (confirm/cancel) use requireAuthedMember = team RBAC. Buyer cookies cannot trigger admin actions.
  • File downloads use signed URLs. Each download issues a fresh 15-min URL — so even if a URL leaks, it expires quickly.
  • ⚠ Shared cookie name with admin/auth-gateway. Buyer auth currently uses the same __session cookie as platform admin/auth-gateway. If a buyer is also a tenant operator on any site in the same browser, the two roles overwrite each other's cookie. Not an auth bypass (Firebase Auth still validates), but a UX collision. Flagged for Plan 3 to consider a separate __dg_session cookie.
