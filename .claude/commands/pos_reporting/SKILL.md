# /pos_reporting — POS Reporting & Analytics Module

You are working on the **Clicker Platform POS Reporting Module**. This system provides analytics and dashboards based on data generated by the `byod_pos` module.

This skill is invoked as `/pos_reporting [action]`.

---

## 1. Architecture Overview

- **Module Directory:** `lib/modules/pos-reporting` (Note the hyphen).
- **Core Logic:** The logic currently heavily relies on client-side or server-side mapping of arrays of raw `POSOrder` objects, passing them through `calculator.ts`.
- **Data Source:** This module rarely writes data; it almost exclusively reads from `sites/{siteId}/modules/byod_pos/orders` and aggregates the results.

---

## 2. Aggregations (`calculator.ts`)

The `calculator.ts` file acts as the single source of truth for business math. If you need a new metric, add a function here.

### Available Summaries

- **`ReportSummary`**: Handles high-level metrics like `totalSales`, `totalOrders`, `averageOrderValue`, and `paymentBreakdown` (how much cash vs. card). It explicitly filters for `completed`, `ready`, or `served` orders.
- **`ItemSalesSummary`**: Iterates through every item within every order to calculate which products are selling best (`quantity` and `revenue` per item).

### Action: `add-report-metric`

If a tenant requests a new metric (e.g., "Total Discounts Given"):

1. Update the `ReportSummary` interface in `calculator.ts`.
2. Update the `calculateReportSummary()` function to `reduce` the `totalDiscount` field from the raw orders.
3. Update the dashboard UI component to render the new metric card.

---

## Common Gotchas

- **Timezones & Date Filtering:** When fetching orders for reporting, the query boundary (`startAt`, `endAt`) must carefully account for the tenant's local timezone. A "Today's Sales" report must start at Midnight local time, not Midnight UTC.
- **Calculation Drift:** Never write inline math like `orders.reduce((sum, o) => sum + o.total, 0)` directly inside UI React components. Always push this logic into `calculator.ts` to ensure the dashboard and exported CSVs use perfectly matched logic.
- **Firestore Read Limits:** For tenants with thousands of orders, pulling all raw documents to the client for client-side aggregation will break cost effectively. For long-term or lifetime metric requests, consider implementing a Server Action that uses Firestore Aggregation Queries (`COUNT`, `SUM`, `AVG`) rather than pulling complete documents.
